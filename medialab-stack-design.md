# medialab-stack

> "코드를 빠르게 생산하는 도구가 아니라, 아이디어를 빠르게 실험하고, 가르치고, 기록하는 도구."

gstack이 스타트업 엔지니어링 팀을 시뮬레이션한다면,
medialab-stack은 **미디어아트 스튜디오의 워크플로우**를 시뮬레이션합니다.

---

## 철학: gstack과 무엇이 다른가

| | gstack | medialab-stack |
|---|---|---|
| **핵심 사이클** | Think → Plan → Build → Review → Test → Ship | Imagine → Sketch → Explore → Refine → Document → Teach |
| **"완료"의 정의** | PR이 머지되고 프로덕션에 배포됨 | 작품이 전시되거나, 수업에서 가르쳐지거나, 논문에 기록됨 |
| **실패의 의미** | 버그, 회귀, 장애 | 흥미롭지 않은 결과 (하지만 이것도 기록할 가치가 있음) |
| **"리뷰"의 기준** | 코드 품질, 보안, 테스트 커버리지 | 미학적 판단, 개념적 정합성, 교육적 명확성 |
| **탐색의 위치** | 계획 이전 단계 (scope를 줄이기 위해) | 핵심 작업 자체 (우연의 발견이 목적) |
| **문서화의 대상** | README, API docs, CHANGELOG | 아티스트 스테이트먼트, 테크니컬 라이더, 워크숍 자료, 논문 |

### 핵심 원칙

1. **탐색은 낭비가 아니다.** 100개의 스케치 중 3개가 작품이 되는 것이 정상이다. 나머지 97개도 카탈로그에 남긴다.
2. **가르치는 것이 이해하는 것이다.** 교육 자료 생성은 부수적 작업이 아니라 창작의 연장선이다.
3. **기술은 표현의 수단이다.** 프레임워크 선택, 최적화, 아키텍처는 미학적 의도를 실현하기 위한 것이다.
4. **기록은 미래의 자산이다.** 오늘의 실험 코드는 내년 수업의 예제가 되고, 3년 후 회고전의 아카이브가 된다.

gstack의 "Think → Ship"이 **선형적 수렴**이라면,
이 사이클은 **나선형 발산과 수렴의 반복**입니다.
Explore 단계에서 예상치 못한 결과가 나오면 Imagine으로 돌아가는 것이 정상입니다.

## 스킬 상세 설계

### Phase 1: IMAGINE — 발상

#### `/imagine` — 컨셉 발상 파트너

gstack의 `/office-hours`에 대응하지만, 소프트웨어 요구사항이 아닌 **예술적 의도**를 파고듭니다.

**역할**: 큐레이터 + 드라마투르그

**작동 방식**:
1. 다섯 가지 질문으로 시작:
   - "이 작품을 경험한 관객이 어떤 상태가 되길 원하는가?" (감각적 목표)
   - "이 아이디어의 핵심 긴장은 무엇인가?" (개념적 프레임)
   - "어떤 매체/기술이 이 긴장을 가장 잘 드러내는가?" (매체 선택)
   - "이것이 기존의 무엇과 다른가?" (포지셔닝)
   - "가장 단순한 형태로 이 아이디어를 실현하면?" (최소 실험 단위)
2. 답변을 바탕으로 **컨셉 문서** 생성:
   - 한 문장 아티스트 스테이트먼트 초안
   - 기술 스택 제안 (p5.js / Processing / oF / TouchDesigner / Max/MSP 등)
   - 3가지 접근 방식 (안전한 선택 / 도전적 선택 / 급진적 선택)
   - 레퍼런스 아티스트/작품 키워드
3. 이 문서는 이후 모든 스킬의 컨텍스트로 사용됨

**출력물**: `CONCEPT.md`

**예시 대화**:
```
You:    /imagine
Claude: 어떤 작품을 구상하고 있나요?

You:    관객의 심박수에 반응하는 제너러티브 비주얼 인스톨레이션.
        심장이 빨리 뛰면 비주얼이 불안정해지고, 
        차분해지면 비주얼이 안정되는 바이오피드백 루프.

Claude: 흥미로운 출발점인데, 한 가지 도발적인 질문을 하겠습니다.
        "차분함 = 안정적 비주얼"이라는 매핑이 너무 직관적이지 않나요?
        
        만약 심박수가 낮아질수록 비주얼이 *더* 불안정해진다면?
        관객이 자기 평온함이 시각적 혼란을 만든다는 역설을 경험하면
        바이오피드백의 의미가 완전히 달라집니다.
        
        세 가지 방향을 제안합니다:
        1. 직관적 매핑 (안전) — 심박수 ↔ 시각적 에너지 비례
        2. 역설적 매핑 (도전) — 평온함이 혼란을, 흥분이 질서를
        3. 시간 지연 매핑 (급진) — 30초 전의 심박수가 지금의 비주얼에
        
        어떤 방향이 끌리세요? 아니면 다른 가능성이 보이시나요?
```

#### `/survey` — 레퍼런스 리서치

**역할**: 리서치 어시스턴트

**작동 방식**:
1. `CONCEPT.md` 또는 사용자 입력 기반으로 관련 레퍼런스 탐색
2. 세 가지 축으로 조사:
   - **선행 작품**: 유사한 컨셉/매체의 미디어아트 작품
   - **기술 레퍼런스**: 관련 라이브러리, 프레임워크, 기법 (논문 포함)
   - **영감 소스**: 다른 분야에서의 유사 접근 (건축, 음악, 과학 등)
3. 각 레퍼런스에 대해:
   - 한 줄 요약
   - 관련성 설명 (이 프로젝트에 왜 중요한지)
   - 기술적 인사이트 (구현에 참고할 점)
4. 웹 검색을 활용하여 최신 레퍼런스 포함

**출력물**: `REFERENCES.md` (프로젝트 루트에 누적)

### Phase 2: SKETCH — 빠른 프로토타입

#### `/sketch` — 즉시 프로토타이핑

**역할**: 숙련된 크리에이티브 코더

**작동 방식**:
1. `CONCEPT.md`를 읽고 (없으면 사용자 설명 기반으로)
2. 대상 프레임워크를 확인:
   - p5.js → HTML 단일 파일
   - Processing → .pde 파일
   - openFrameworks → ofApp.h/cpp
   - TouchDesigner → Python 스크립트 + 노드 설명
   - Max/MSP → 패치 설명 + JavaScript
   - Three.js → HTML 단일 파일
   - Tone.js → HTML 단일 파일 (오디오)
   - GLSL → 셰이더 파일 + 호스트 코드
3. **최소 동작 코드**를 생성 (실행 가능한 상태)
4. 핵심 파라미터를 변수로 추출하여 쉽게 조절 가능하게
5. 코드 상단에 간략한 사용법 주석

**핵심 차별점**: gstack의 `/ship`이 "완성된 코드"를 목표로 한다면,
`/sketch`는 **"실행 가능한 최소 실험"**을 목표로 합니다.
완벽할 필요 없음. 빠르게 눈으로 보는 것이 핵심.

**예시**:
```
You:    /sketch 심박수 데이터를 받아서 파티클 시스템을 제어하는 p5.js 스케치.
        WebSocket으로 심박수 데이터 수신. BPM이 높을수록 파티클 속도와 수 증가.

Claude: [CONCEPT.md가 있으면 참조, 없으면 바로 진행]
        [p5.js 단일 파일 생성 — WebSocket 연결, 파티클 클래스, 
         BPM → 파라미터 매핑, 즉시 실행 가능]
        
        핵심 파라미터가 상단에 있으니 바로 수정해 보세요:
        - PARTICLE_COUNT_BASE: 기본 파티클 수
        - BPM_SENSITIVITY: BPM 변화에 대한 민감도
        - COLOR_PALETTE: 색상 팔레트 배열
        
        /vary로 색상이나 움직임 패턴의 변형을 만들 수 있습니다.
```

#### `/vary` — 변형 생성

**역할**: 디자인 탐험가

**작동 방식**:
1. 기존 스케치 코드를 읽고
2. 변형 축을 제안 (또는 사용자 지정):
   - 색상/팔레트 변형
   - 움직임/물리 법칙 변형
   - 스케일/밀도 변형
   - 알고리즘 변형 (노이즈 종류, 매핑 함수 등)
3. N개의 변형을 별도 파일로 생성 (기본 3개)
4. 각 변형에 어떤 파라미터를 어떻게 바꿨는지 주석

**출력물**: `sketches/[name]-v1.js`, `sketches/[name]-v2.js`, `sketches/[name]-v3.js`

### Phase 3: EXPLORE — 탐색과 실험

#### `/experiment` — 체계적 실험

**역할**: 연구원

**작동 방식**:
1. 기존 스케치에서 탐색할 파라미터 공간을 정의
2. 파라미터 조합 매트릭스 생성
3. 각 조합을 실행/캡처하는 스크립트 생성 (가능한 경우)
4. 결과를 구조화된 실험 로그로 기록

**출력물**: `experiments/[date]-[name]/` 디렉토리
- `experiment-log.md` — 파라미터, 결과, 관찰 기록
- 각 변형의 코드 스냅샷
- 스크린샷/캡처 스크립트 (프레임워크가 지원하는 경우)

**핵심 차별점**: `/vary`가 "이런 변형도 있어요"라면,
`/experiment`는 "이 파라미터 범위를 체계적으로 탐색하고 결과를 기록합시다"입니다.

---

#### `/glitch` — 우연의 발견

**역할**: 트릭스터

**작동 방식**:
1. 기존 스케치의 규칙을 의도적으로 깨뜨림:
   - 데이터 타입 오용 (float을 int로, 배열 인덱스 초과 등)
   - 물리 법칙 위반 (중력 반전, 시간 역행)
   - 입력/출력 뒤섞기 (오디오를 비주얼로, 좌표를 색상으로)
   - 해상도/스케일 극단적 변형
2. 각 "글리치"를 별도 파일로, 무엇을 깨뜨렸는지 주석
3. 흥미로운 결과에 대해 "이것을 의도적 미학으로 발전시키려면?" 제안

**주의**: 이 스킬은 **데이터를 파괴하지 않습니다**. 원본은 항상 보존하고 복사본에서 실험합니다.

---

#### `/digest` — 논문/기술 요약

**역할**: 리서치 어시스턴트

**작동 방식**:
1. 논문 URL, PDF, 또는 주제 키워드를 입력받음
2. 세 가지 관점으로 분석:
   - **핵심 아이디어**: 한 문단 요약
   - **구현 노트**: 미디어아트 프로젝트에 적용할 때 필요한 기술적 세부사항
   - **영감 포인트**: 이 기법을 예술적으로 재해석할 수 있는 방향
3. 관련 코드 예시가 있으면 대상 프레임워크로 번역
4. `REFERENCES.md`에 자동 추가

**예시**:
```
You:    /digest https://arxiv.org/abs/xxxx.xxxxx
        이 논문의 optical flow 기법을 p5.js에서 어떻게 쓸 수 있을지

Claude: [논문 분석]
        핵심: ...
        구현 노트: WebGL 셰이더로 실시간 처리 가능. 
        p5.js에서는 createGraphics + shader()로 구현.
        [최소 구현 코드 스니펫]
        
        영감 포인트: 이 알고리즘의 "오류"가 오히려 회화적 효과를 만듦.
        의도적으로 해상도를 극단적으로 낮추면 인상파풍의 결과물 가능.
```

### Phase 4: REFINE — 정제

#### `/critique` — 미학적 리뷰

**역할**: 미술 비평가 패널 (형식주의자 + 개념주의자 + 기술 비평가)

**작동 방식**:
1. 코드와 `CONCEPT.md`를 함께 읽고
2. 세 가지 관점에서 평가:
   - **형식적 분석**: 구성, 색채, 리듬, 공간 사용, 시각적 긴장
   - **개념적 정합성**: 의도한 컨셉이 매체/기술 선택에 잘 반영되었는가
   - **관객 경험**: 인터랙션 디자인, 접근성, 경험의 흐름
3. 각 항목을 0-10 점수로 평가하되, **"10이 어떤 상태인지"를 구체적으로 설명**
4. 점수를 올리기 위한 구체적 제안 (코드 수정 방향 포함)

**핵심 차별점**: gstack의 `/review`가 "버그와 코드 품질"을 보는 반면,
`/critique`는 **"이 코드가 만들어내는 경험이 의도에 부합하는가"**를 봅니다.

---

#### `/optimize` — 퍼포먼스 최적화

**역할**: 테크니컬 아티스트

**작동 방식**:
1. 코드 분석 후 병목 지점 식별:
   - 프레임 레이트 (타겟: 보통 60fps, VR은 90fps)
   - 메모리 사용량 (장시간 전시의 메모리 누수)
   - 시작 시간 (전시 오프닝 전 빠른 부팅)
   - GPU/CPU 사용률
2. 최적화 제안을 **미학적 영향과 함께** 설명:
   - "파티클을 10,000개에서 5,000개로 줄이면 30fps → 60fps,
     하지만 밀도감이 줄어듦. 대안: 인스턴스드 렌더링으로 10,000개 유지"
3. 장시간 운영 안정성 체크:
   - 메모리 누수 패턴
   - 에러 복구 로직
   - 자동 재시작 스크립트

---

#### `/install` — 전시 배포

**역할**: 전시 기술 감독

**작동 방식**:
1. 프로젝트를 분석하여 배포 체크리스트 생성:
   - **하드웨어 요구사항**: 컴퓨터 스펙, 프로젝터/디스플레이, 센서, 오디오
   - **소프트웨어 셋업**: OS 설정, 자동 시작, 화면보호기 비활성화, 에러 복구
   - **네트워크**: 필요한 포트, 외부 연결, 오프라인 동작 가능 여부
   - **현장 조건**: 조명, 공간, 전원, 케이블 경로
2. 테크니컬 라이더 문서 생성 (갤러리/페스티벌 제출용)
3. 자동 시작 스크립트 생성 (OS별: macOS launchd, Linux systemd, Windows Task Scheduler)
4. 원격 모니터링 설정 가이드

**출력물**: `TECHNICAL-RIDER.md`, `scripts/autostart/` 디렉토리

---

### Phase 5: DOCUMENT — 기록

#### `/document` — 프로젝트 문서화

**역할**: 아카이비스트 + 큐레이터

**작동 방식**:
1. `CONCEPT.md`, 코드, `REFERENCES.md`를 종합하여:
   - **아티스트 스테이트먼트** (100단어 / 300단어 / 500단어 버전)
   - **기술 설명** (비기술자용 / 기술자용)
   - **프로젝트 페이지** (포트폴리오 웹사이트용 마크다운)
   - **전시 캡션** (갤러리 벽면용 간결한 텍스트)
2. 국문/영문 모두 생성 (미디어아트 분야는 이중언어가 필수적)
3. 기존 문서가 있으면 코드 변경사항 반영하여 업데이트

**출력물**: `docs/` 디렉토리에 체계적으로 저장

---

#### `/archive` — 코드 정리 및 카탈로그

**역할**: 디지털 아카이비스트

**작동 방식**:
1. 프로젝트 디렉토리를 스캔하여:
   - 실험 코드, 최종 코드, 유틸리티 코드 분류
   - 미사용 파일 식별
   - 의존성 정리 (package.json 등)
2. 디렉토리 구조 제안:
   ```
   project/
   ├── src/           ← 최종 실행 코드
   ├── sketches/      ← 실험/스케치 (날짜별)
   ├── assets/        ← 미디어 파일
   ├── docs/          ← 문서
   ├── tools/         ← 유틸리티/스크립트
   └── archive/       ← 더 이상 사용하지 않지만 보존할 코드
   ```
3. 각 스케치/실험에 메타데이터 태그 추가:
   - 날짜, 사용 기술, 관련 컨셉, 상태(실험/완성/폐기)
4. `CATALOG.md` 생성 — 모든 스케치의 인덱스

---

#### `/cite` — 참고문헌 관리

**역할**: 학술 서지 관리자

**작동 방식**:
1. `REFERENCES.md`의 레퍼런스를 정형화된 인용 형식으로 변환
2. 지원 형식: BibTeX, APA, MLA, Chicago, ACM
3. 논문 URL/DOI에서 자동으로 서지정보 추출
4. 중복 제거 및 관련성 순 정렬
5. 학회 제출용 참고문헌 섹션 생성 (NIME, SIGGRAPH, Ars Electronica 등의 형식)

### Phase 6: TEACH — 교육

#### `/workshop` — 워크숍 자료 생성

**역할**: 교육 설계자

**작동 방식**:
1. 기존 코드를 입력으로 받아:
   - **슬라이드 구조** (발표 순서, 각 슬라이드의 핵심 메시지)
   - **핸즈온 가이드** (학생용 step-by-step 튜토리얼)
   - **코드 스니펫** (단계별로 나뉜 점진적 코드)
   - **체크포인트** (각 단계가 끝난 후 보여야 할 결과)
2. 난이도 설정 가능: 입문 / 중급 / 고급
3. 시간 설정 가능: 1시간 / 2시간 / 반일 / 종일
4. 대상 플랫폼 설정: p5.js 에디터 / Processing IDE / 로컬 개발환경

**출력물**:
- `workshop/slides.md` (프레젠테이션 구조)
- `workshop/handout.md` (학생 배포용)
- `workshop/steps/` (단계별 코드 파일)
- `workshop/solutions/` (최종 완성 코드)

**예시**:
```
You:    /workshop 이 파티클 시스템 코드를 2시간짜리 중급 워크숍 자료로 만들어줘.
        대상: p5.js 기초는 아는 대학생, p5.js 웹 에디터 사용.

Claude: [코드 분석 → 학습 목표 추출 → 단계 분해]
        
        워크숍: "데이터 기반 파티클 시스템"
        시간: 2시간 (15분 소개 + 4x25분 실습 + 15분 마무리)
        
        Step 1: 기본 파티클 클래스 (25분)
        Step 2: 물리 시뮬레이션 추가 (25분)  
        Step 3: 외부 데이터 연결 (25분)
        Step 4: 시각적 커스터마이징 (25분)
        
        [각 단계별 시작 코드, 완성 코드, 학생 가이드 생성]
```

---

#### `/explain` — 코드 해설

**역할**: 교수 조교

**작동 방식**:
1. 코드를 입력받아 **교육적 관점에서** 주석 달기:
   - 단순 "이 줄은 X를 합니다"가 아닌
   - "왜 여기서 noise()를 쓰는지", "이 매핑의 미학적 효과는 무엇인지"
2. 개념별 설명 블록:
   - 수학적 배경 (삼각함수, 벡터, 노이즈 등)
   - 시각적 결과와 코드의 관계
   - "이 부분을 바꾸면 무엇이 달라지는지"
3. 어려운 개념에 대한 직관적 비유
4. 원본을 손상하지 않고 주석이 달린 버전을 별도 생성

**출력물**: `[filename]-annotated.[ext]`

---

#### `/exercise` — 과제/연습문제 생성

**역할**: 교육과정 설계자

**작동 방식**:
1. 주제 또는 기존 코드를 기반으로
2. 난이도별 과제 생성 (보통 3단계):
   - **기초**: 파라미터 수정으로 결과 관찰 (코드 이해)
   - **응용**: 기존 코드에 기능 추가 (기법 적용)
   - **창작**: 동일 기법으로 완전히 새로운 작품 (자기 표현)
3. 각 과제에 포함:
   - 명확한 요구사항
   - 시작 코드 (스켈레톤)
   - 힌트 (접힌 상태로)
   - 평가 기준 (기술적 + 미학적)
   - 예시 결과물 설명

---

#### `/syllabus` — 커리큘럼 설계

**역할**: 커리큘럼 아키텍트

**작동 방식**:
1. 과목 정보를 입력받아:
   - 과목명, 학기 길이 (주 단위), 주당 시수
   - 학생 수준 (비전공 입문 / 전공 중급 / 대학원)
   - 사용 도구/프레임워크
   - 학습 목표
2. 주차별 커리큘럼 생성:
   - 각 주의 주제, 학습 목표, 수업 활동
   - 필수/선택 과제
   - 참고 자료
   - 중간/기말 프로젝트 일정
3. 기존에 만든 `/workshop` 자료나 `/exercise` 과제와 연결
4. 학기 전체의 난이도 커브 시각화

**출력물**: `syllabus/SYLLABUS.md` + 주차별 디렉토리 구조

## 기술 설계

### gstack의 구조를 차용하되

- **SKILL.md 기반**: 각 스킬은 독립된 마크다운 파일로 정의
- **슬래시 명령**: Claude Code에서 `/sketch`, `/vary` 등으로 호출
- **컨텍스트 체인**: `CONCEPT.md` → 다른 스킬들이 자동으로 참조
- **비파괴적**: 원본 코드를 수정하지 않고 항상 새 파일/디렉토리에 결과 생성
- **프레임워크 감지**: 프로젝트의 파일 구조를 보고 p5.js/Processing/oF 등을 자동 감지

### 프로젝트 표준 구조

```
my-project/
├── CONCEPT.md              ← /imagine이 생성, 모든 스킬의 컨텍스트
├── REFERENCES.md           ← /survey, /digest가 누적
├── CATALOG.md              ← /archive가 생성/업데이트
├── TECHNICAL-RIDER.md      ← /install이 생성
├── src/                    ← 메인 실행 코드
├── sketches/               ← /sketch, /vary의 결과물
│   ├── 2026-03-30-particles-v1.js
│   ├── 2026-03-30-particles-v2.js
│   └── ...
├── experiments/            ← /experiment의 결과물
│   └── 2026-03-30-bpm-sweep/
│       ├── experiment-log.md
│       └── ...
├── docs/                   ← /document의 결과물
│   ├── artist-statement-ko.md
│   ├── artist-statement-en.md
│   └── ...
├── workshop/               ← /workshop의 결과물
│   ├── slides.md
│   ├── handout.md
│   ├── steps/
│   └── solutions/
├── syllabus/               ← /syllabus의 결과물
└── archive/                ← /archive가 정리한 이전 코드
```

### 멀티 호스트 지원

gstack과 동일하게 Claude Code를 기본 호스트로 하되,
Codex, Cursor 등에서도 동작하도록 SKILL.md 표준을 따름.

## 부록: 예시 시나리오

### 시나리오 1: 전시 준비 (4주)

```
Week 1: /imagine → 컨셉 정립
        /survey → 레퍼런스 수집
        /sketch → 첫 프로토타입 3개

Week 2: /vary → 각 프로토타입에서 5개 변형
        /experiment → 파라미터 공간 탐색
        /critique → 중간 리뷰

Week 3: /optimize → 60fps 안정화, 메모리 누수 제거
        /install → 테크니컬 라이더 작성
        /document → 아티스트 스테이트먼트

Week 4: /install → 자동 시작 스크립트, 현장 셋업
        /archive → 프로젝트 정리
```

### 시나리오 2: 워크숍 준비 (1주)

```
Day 1-2: /imagine → 워크숍 컨셉 (무엇을 가르칠 것인가)
         /sketch → 데모 코드 작성

Day 3-4: /explain → 코드 해설 버전 작성
         /workshop → 슬라이드 + 핸즈온 자료

Day 5:   /exercise → 과제 3개 (기초/응용/창작)
         최종 리허설
```

### 시나리오 3: 학기 수업 준비

```
/syllabus → 16주 커리큘럼 생성
각 주마다:
  /sketch → 수업 예제 코드
  /explain → 주석이 달린 학습 버전
  /exercise → 주간 과제
  /workshop → 수업 슬라이드
```
